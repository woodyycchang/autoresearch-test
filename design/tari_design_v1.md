# TARI v1 — Transcript-Anchored Rigorous Ideation

**Author:** Claude (Opus 4.7), branch `claude/alternative-niche-mining-pipeline-IVlwA`.
**Date:** 2026-05-21.
**Status:** Design doc — Phase 1 of new project branch alongside `niche-mining-autoresearch`.

---

## 0. Why a fundamentally new pipeline?

### 0.1 Acknowledgement of niche-mining saturation

`niche-mining-autoresearch` v1 through v20 ran N=1071 rounds across 39 epochs with **0 substantive PASS** (definition: candidate that survives step 10 mechanical hit + step 13.5 adversarial + step 14.6 external collision + would survive a real arXiv lit-search outside the synthesized detection layer). 22 candidates currently sit in INVESTIGATIVE_SURVIVING (the "neither PASS nor explicit FAIL" bin), but every single one was generated by Claude sampling from its own coverage of mathematics × ML architecture analogies, and the detector chain that lets them through (step 14.6) is **itself synthesized in main-context-direct mode** — it cannot see anything Claude could not have seen during step 05 generation.

That structural fact is the root of v20's diagnostic: the pipeline has world-model and user-model components, but it has no source of candidates that are *not* drawn from Claude's own analogy coverage. v20 added a self-model layer to make the generator introspect on its own state; this is useful for understanding *why* mining saturates, but it does not fix the input bottleneck.

**Continued mining will not produce a new substantive PASS.** Adding more upgrade layers (v21 world-model refinement, v22 retrieval, v23 multi-agent debate) will at best produce more INVESTIGATIVE_SURVIVING candidates that all fail real external verification for the same reason: they're sampled from a saturated distribution.

### 0.2 What TARI changes

TARI flips the input source. Instead of asking Claude "find a niche somewhere in your coverage of AI × mathematics", TARI takes a **specific frontier-research transcript** (a colloquium talk or invited talk at a top venue, recorded 2025-2026), decomposes it into snippets, extracts atomic claims/primitives/negative-results from those snippets, brainstorms cross-atom combinations, and produces candidates that are **source-traceable to specific transcript lines.**

The key structural difference is provenance. In mining, "ANCHOR_R843 has yield 3/3" is a statement about the pipeline's internal state — there is no external referent. In TARI, "atom A_007 lives at lines 142-156 of belinda_li_self_models.txt" is a statement about an external artifact that an external reviewer can look up.

This does not make TARI immune to Claude's coverage — atom extraction is still Claude reading the transcript, brainstorming is still Claude — but every candidate carries an externally-checkable claim: *"my atoms came from these specific lines of this specific transcript."* If those lines do not say what the candidate claims they say, the candidate fails self-model audit before it ever reaches external verification.

### 0.3 Why this is "rigorous ideation" not just "another generator"

Three rigor mechanisms:

**(a) Source-anchored audit.** Every candidate must answer Belinda Li's three self-model questions:
- Q1: What internal activations (= which transcript atoms) represent the claim?
- Q2: What internal components (= which combination operators) did I use to produce this?
- Q3: What input components (= which verbatim transcript lines) drove this?

If Q3 cites lines that do not exist or do not say what the candidate claims, the candidate fails. The audit is mechanical: line numbers are checked against the transcript file.

**(b) External verification reuses the existing detector chain.** Step 06 (lit-search), step 13.5 (adversarial), step 14.6 (external collision) all run on TARI candidates exactly as they ran on mining candidates. TARI does not weaken any existing filter. If TARI is just dressed-up mining, it will saturate to 0 PASS at the same rate.

**(c) Empirical test stage.** Candidates that survive external verification must propose an empirical test that is (i) implementable with a research-lab budget, (ii) has a falsifiable prediction, (iii) is not already published. This is the Stage 1.5 gate carried over from the niche-mining design but applied per-candidate.

### 0.4 Honest framing

TARI is not guaranteed to find a substantive PASS in one epoch. It is guaranteed to produce candidates that are *traceable to a different distribution* than mining candidates — specifically, the distribution of claims a frontier researcher made in their own talk. Whether that distribution contains substantive niches that Claude can recognize and external verification cannot disprove is an empirical question. The honest prior, given mining's 0/1071, is that one TARI epoch will most likely also produce 0 PASS but with **different failure modes**, and surfacing those failure modes is the scientific point.

## 1. Architecture

```
INPUT_TRANSCRIPT (text, line-numbered)
        |
        v
[snippet_decomposer.py]
        |  Produces N snippets with (start_line, end_line, verbatim_text, boundary_justification)
        v
SNIPPETS/snippet_001.json ... snippet_N.json
        |
        v
[atom_extractor.py]
        |  Per-snippet, extract atoms typed as one of:
        |    - PRIMITIVE          (a named mechanism the speaker invokes, e.g. "associative scan")
        |    - MECHANISM_CLAIM    (a causal/structural claim, e.g. "LMs build latent world models via probing-recoverable activations")
        |    - NEGATIVE_RESULT    (a thing that failed or was explicitly ruled out)
        |    - METRIC             (a measurement the speaker reports)
        |    - OPEN_QUESTION      (a thing the speaker said is unsolved)
        |  Each atom carries (id, type, snippet_id, line_span, verbatim_quote, gloss)
        v
ATOMS/atom_001.json ... atom_M.json
        |
        v
[brainstorm_engine.py]
        |  Cross-atom combination — for each candidate proposal, record:
        |    - candidate_id
        |    - combined_atom_ids (list, must be >=2 from distinct snippets)
        |    - combination_operator (one of: ANALOGIZE, INVERT, COMPOSE, GENERALIZE, RESTRICT, CONTRAST)
        |    - claim                (1-2 sentences)
        |    - why_novel_vs_speaker (why this is not what the speaker said)
        |    - why_potentially_useful (1 sentence)
        v
CANDIDATES/candidate_001.json ... candidate_K.json
        |
        v
[self_model_audit.py]
        |  Per candidate, answer Belinda's 3 questions:
        |    Q1: which atoms represent the activations driving this claim?
        |    Q2: which combination operators were used (and on which atoms)?
        |    Q3: which verbatim transcript lines anchor this (must cite line numbers)?
        |  MECHANICAL CHECKS:
        |    - all cited atom_ids must exist in ATOMS/
        |    - all cited line numbers must fall in cited snippets' line spans
        |    - the verbatim_quote field of each cited atom must literally appear in the transcript
        |  Candidate fails audit if any check fails.
        v
AUDIT/self_model_audit.json  (per-candidate verdicts)
        |
        v
[external_verifier.py]
        |  For each audit-surviving candidate, run the existing detector chain:
        |    - step 06 lit-search (real web_search this time, not synthesized)
        |    - step 13.5 adversarial spec
        |    - step 14.6 external collision check
        |  Candidate fails if step 14.6 produces EXTERNAL_COLLISION or if step 06 reports >=2 keyword hits.
        v
VERDICT/external_verification.json
        |
        v
[orchestrator.py]  prints run summary, writes runs/run_NNN/summary.md
```

### 1.1 Line-numbered transcript convention

The input transcript is stored verbatim. snippet_decomposer.py reads the file once and produces snippets indexed by 1-based line numbers. Every downstream artifact references line numbers, not byte offsets, because line numbers are stable under formatting (and the input transcripts may be reformatted by upstream).

### 1.2 Atom IDs

Atom IDs are deterministic: `ATOM_<snippet_id>_<atom_index_within_snippet>`. E.g. `ATOM_S003_02` = the 2nd atom in snippet 003. This makes the brainstorm output's atom-ID list machine-checkable.

### 1.3 Candidate IDs

`CAND_<run_id>_<candidate_index>`. E.g. `CAND_001_007`.

### 1.4 Why combination operators are typed

Free-form "combine these atoms" produces unbounded creativity but no audit trail. By restricting to 6 named operators (ANALOGIZE, INVERT, COMPOSE, GENERALIZE, RESTRICT, CONTRAST), we make the brainstorm step's output reproducible and let the audit step verify: did the candidate actually do an INVERT of atom A_005, or did it just invent something?

## 2. Why TARI is fundamentally different from mining

| Aspect | niche-mining (v1-v20) | TARI v1 |
|---|---|---|
| Input distribution | Claude's coverage of math×ML analogies | Specific transcript's claim set |
| Candidate provenance | Internal pipeline state (anchor, slot, frontier_seed) | External artifact (transcript file, line numbers) |
| Audit referent | None — lineage claims are narrative | Mechanical — line numbers checked against file |
| What an external reviewer can verify before reading the candidate | Nothing | The atoms exist in the transcript |
| Saturation behavior | Each round samples a slightly different point in the same distribution | Each new transcript opens a new distribution |
| Failure mode if Claude has full coverage | Indistinguishable from random over a saturated set | Candidate must still be defensible against the transcript's own claims |

The crucial distinction: in mining, "this niche is novel" means "Claude couldn't find it in step 06's synthesized search." In TARI, "this niche is novel" means "the transcript speaker did not make this exact claim, but it combines their atoms in a way they did not." The former is unprovable from outside; the latter is mechanically checkable.

## 3. Honest predicted failure modes

These are the things most likely to break TARI v1. Calling them out so the run produces useful evidence about which broke.

### 3.1 Atom granularity wrong

If atoms are too coarse (one atom per snippet), brainstorm has nothing to combine. If too fine (one atom per sentence), atoms are not distinct claims and combinations are trivial restatements. **Prediction:** v1 will produce 20-60 atoms total from a ~10000-word transcript; many will overlap. The audit step will fail many candidates because cited atoms turn out to be near-duplicates. **Diagnostic to surface in summary.md:** average atom uniqueness (Jaccard over content_words between atoms).

### 3.2 Brainstorm hallucinates atoms

The brainstorm step is the most dangerous: Claude could combine atoms A_007 and A_023 by *implicitly* invoking a fact that is not in either atom. The candidate then claims "uses atoms A_007 + A_023" but the actual reasoning is "atoms A_007 + A_023 + a hidden fact from Claude's pre-training." **Prediction:** ~30-50% of v1 candidates will have at least one cited line number that doesn't actually contain the claim attributed to it. **Diagnostic:** self-model audit fail rate (target: real, non-zero — if 0%, audit is too lax).

### 3.3 Self-model fabricates traceability

Belinda's Q3 (verbatim lines) is the strongest check, but Claude could pass it by *cherry-picking* lines that contain the right keywords without containing the claim. **Prediction:** Some candidates will pass mechanical line-citation check but fail a more semantic "does the cited passage actually entail what the candidate claims" check. **Diagnostic:** spot-check on each surviving candidate in summary.md (manual section).

### 3.4 External verifier still synthesizes

Step 06 / 13.5 / 14.6 in the existing pipeline run in main-context-direct mode (synthesized). If TARI reuses them as-is, candidates that pass them have only passed Claude's simulation of an external check. **Mitigation in v1:** external_verifier.py will do *one real web_search per surviving candidate* (the WebSearch tool is available via ToolSearch), separate from the synthesized step 06 / 14.6. If real and synthesized disagree on any candidate, that disagreement is flagged in summary.md.

### 3.5 Transcripts contain only one speaker's view

A single transcript is by definition a coverage subset of the field. TARI v1 with one transcript will produce candidates biased toward that speaker's framing. **Mitigation:** the design is per-transcript; subsequent runs use different transcripts. v1 epoch 1 establishes whether the pipeline works on one transcript; cross-transcript brainstorm is v2.

### 3.6 Brainstorm produces candidates that are just the speaker's claim restated

Most likely failure: candidates are mild rephrasings of atoms, not novel combinations. **Mitigation:** the `why_novel_vs_speaker` field is required and externally checked by self-model audit — Q3 must cite the speaker's verbatim claim AND the candidate must differ from it in a way that is articulable. A candidate that says "differs from speaker because [restates speaker]" fails.

## 4. Success criteria for v1 epoch 1

**Minimal success:** ≥1 candidate passes self-model audit (all 3 Belinda questions, mechanical line-citation check) AND passes external verification (step 06 keyword threshold < 2 AND step 14.6 SURVIVES) AND has a documented Stage 1.5-style empirical test in its candidate JSON.

**Stretch goals:**
- ≥3 candidates pass self-model audit (shows audit is not vacuous)
- ≥1 candidate has a real web_search keyword count < synthesized step 06's count (shows real verification finds something synth missed — or matches it, either is informative)
- ≥1 candidate's `why_novel_vs_speaker` is sustainable under a 3-sentence challenge ("name one paper/talk that makes this exact claim")

**Honest non-criteria:** v1 does *not* need to produce a "novel discovery" in any real sense. It needs to demonstrate that the pipeline *can* produce a source-traceable, mechanically-auditable, externally-verified candidate that survives an empirical-test specification. The hard question of whether that candidate is *actually new* requires real arXiv / Google Scholar search beyond the WebSearch tool, which is out of scope for v1 but is the natural Stage 1.6 extension.

**Failure-mode success:** even if 0 candidates pass everything, v1 epoch 1 succeeds scientifically if the summary.md surfaces *which* of §3's predicted failure modes dominated. That is information mining could not produce.

## 5. Why the Belinda Li transcript was chosen as INPUT_TRANSCRIPT_001

Three reasons:

**(a) Self-evident match.** The talk is explicitly about self-models in language agents — the same framework TARI's audit step uses. Bootstrapping a self-model-audit pipeline on a self-model talk creates a closed loop: if TARI's audit cannot handle this transcript, it cannot handle any transcript.

**(b) Frontier content (2025-2026).** Belinda Li's published work and the talk's references (Anthropic Opus 4.6 audit, OpenAI self-model papers) place the content firmly in 2025-2026 frontier AI research. This is the regime where niche-mining saturated — if TARI can extract source-traceable candidates from this regime, the comparison to mining is direct.

**(c) Rich atom structure.** The talk has explicit world-model / user-model / self-model sections, each with named techniques (probing, activation patching, GATE, OPEN, self-explanation training). That structure should let atom_extractor.py find 30+ atoms across 3-5 thematic clusters, providing the substrate for non-trivial brainstorm combinations.

The choice is documented in `tari/inputs/INPUT_TRANSCRIPT_001_provenance.md`.

## 6. Comparison to existing niche-mining components — what TARI reuses vs. replaces

| Mining component | TARI v1 disposition |
|---|---|
| Step 05 anchor-local sampling | Replaced by atom_extractor + brainstorm_engine |
| Step 05.5 cascade (FTS/KCD/anti-R279) | Not used in v1 (TARI's audit is source-based) |
| Step 05.6 learned verifier | Not used in v1 |
| Step 05.7 self-model (v20) | Reframed as self_model_audit.py with Belinda's Q1/Q2/Q3 questions |
| Step 06 web_search | REUSED (and augmented with one real WebSearch per surviving candidate) |
| Step 07 keyword threshold | REUSED |
| Step 10 mechanical verdict | REUSED |
| Step 11 / 12 audit | Reframed as part of self-model audit |
| Step 13.5 adversarial spec | REUSED |
| Step 14 / 14.5 / 14.6 | REUSED |

The detector chain (06 through 14.6) is treated as an *external verification layer* that TARI plugs candidates into. TARI's contribution is upstream: it changes where candidates come from.

## 7. What v1 does *not* attempt

- Cross-transcript brainstorming (multiple transcripts at once). v2.
- Learning to weight atoms by speaker authority or atom novelty. v2+.
- Multi-agent debate over candidates. Explicit non-goal — design philosophy is source-traceable single-author candidates.
- A new external verification layer. Honest position: reuse the existing one; TARI's value is in the upstream change.
- Continuous learning across TARI epochs. Each epoch is self-contained.

## 8. Open design questions (documented, not resolved in v1)

- **Atom typing precision.** v1 has 5 atom types. Should there be a sub-type for METHOD vs PRIMITIVE? Decide based on whether atom_extractor produces a high mis-classification rate in epoch 1.
- **Operator granularity.** 6 operators may be too few. After epoch 1, examine whether candidates cluster on 2-3 operators (suggesting we need more) or use all 6 evenly (suggests granularity is right).
- **Audit semantic check.** v1's audit only checks that cited lines exist and contain the verbatim quote. It does not check entailment. Adding entailment requires an LLM call per audit, which raises cost and re-introduces Claude-coverage bias. Hold off until v1 data tells us whether mechanical citation passes too easily.

---

## 9. Honest deviation policy (carried from niche-mining)

Max 5 synthesized Agent spawns per epoch. Real WebSearch is reserved for the external_verifier.py step, and is run once per audit-surviving candidate (expected 1-5 candidates / run, so 1-5 real searches total per epoch). All other pipeline steps run in main-context-direct mode.

---

## 10. Summary

TARI v1 is a fundamentally different pipeline from mining: source-grounded ideation, not Claude-coverage sampling. Its rigor comes from external artifact provenance (transcript line numbers) and a 3-question self-model audit (Belinda Li's framework). It reuses the existing detector chain as an external verification layer. It is not guaranteed to find a substantive PASS in one epoch, but it is guaranteed to produce candidates whose failure modes are *different* from mining's, and that is the scientific contribution v1 sets out to demonstrate.
